> the Stacks: extensive ideation on deep structural fixes and larger expansion
> bets that the existing planning documents do **not** already contain. It was
> produced from a fresh, full read of the source tree, tests, CI, and audit
> artifacts on `main` (HEAD `a19142c`), plus the research pass on the
> `research-panel-and-roadmap` branch.

**Rule followed throughout: NET-NEW only.** Where an idea extends an existing
roadmap item, the existing ID/section is cited and the idea states what it adds
*beyond* it. Where an idea would collide with a guardrail (read-only sources; no
egress of reading data; no Goodreads/gatekept catalogs; sourced descriptors,
never author labels), the tension is named rather than papered over.

## Honest framing

These are **ideas for evaluation, not commitments**. They were generated by one
deep code-reading pass, without new user conversations, without running the
system against a real library, and without consulting the data stewards whose
terms several ideas depend on. Some observations (e.g. the demo-candidate wiring
in `app/view.py`, the missing md5 join in `ingest/unify.py`) are verifiable in
code today; impact estimates and horizon placements are judgment calls and say
so. Nothing here should ship without the same gate discipline the repo already
practices: auto-gates for what is checkable, dated committed artifacts for what
requires human judgment, and honest deferral for everything else.
